# Perfoboard Layout Optimizer

## Назначение

Этот skill превращает описание схемы (детали + корпуса + соединения) в компактную раскладку на макетной плате с сеткой отверстий, минимизируя занимаемую площадь и уменьшая пересечения соединений.

## Когда использовать

- Пользователь даёт размеры платы (например 20×40 отверстий) и список деталей в корпусах с выводами
- Пользователь описывает соединения (нетлист: какие пины должны быть спаяны)
- Нужно получить координаты каждой ножки в отверстиях, ориентацию элементов и план соединений на нижней стороне

## Входные данные, которые нужно запросить/нормализовать

1) Размер платы:
- width: число отверстий по X (целое)
- height: число отверстий по Y (целое)
- (опционально) запрещённые отверстия/зоны: список координат или прямоугольников

2) Список компонентов (BOM) в виде таблицы/списка:
- ref: R1, U1, C3…
- package: тип корпуса (должен совпадать с package-правилами в .trae/rules)
- pins: список имён пинов (например ["1","2"] или ["E","B","C"])
- (опционально) фиксированные требования: нельзя поворачивать, нужно у края, нужен зазор вокруг, полярность и т. п.

3) Соединения (нетлист):
- nets: словарь netName -> список "ref.pin"
- или список соединений вида: ["U1.1 - R1.1 - C1.+", ...]

4) Предпочтения пайки/трассировки:
- допускаются ли перемычки (джамперы) на верхней стороне
- допускаются ли провода на нижней стороне с пересечениями (по умолчанию: избегать, но при необходимости разрешать с джампером)
- предпочтительные “шины”: GND, VCC (если есть)

## Оптимизационная цель (cost function)

Минимизировать лексикографически (в порядке важности):
1) Валидность: отсутствие конфликтов отверстий, соблюдение корпусных правил, все пины размещены
2) Площадь: площадь ограничивающего прямоугольника всех занятых отверстий (bboxArea)
3) Проводимость: суммарная длина соединений на плате (Manhattan по сетке)
4) Пересечения: число пересечений/конфликтов трасс (штраф, высокий)
5) Технологичность: меньше “скачков” проводов, удобные шины, минимальные острые петли

## Процесс (алгоритм)

### 1) Парсинг и проверка ограничений
- Привести все pin-обозначения к "ref.pin"
- Проверить: каждый вывод каждого компонента должен иметь уникальное отверстие
- Сопоставить package каждого компонента с соответствующим rule-файлом; извлечь footprint-модель: какие отверстия занимает, какие перекрывает корпус, допустимые изгибы

### 2) Построение модели размещения
- Представить плату как сетку (x=1..W, y=1..H)
- Для каждого компонента с footprint:
  - сгенерировать варианты ориентации (0/90/180/270 или ограниченные правилами)
  - сгенерировать допустимые “формы” (если правила допускают отгиб выводов)
  - для каждого варианта определить:
    - множество отверстий, куда вставляются пины (pinHoles)
    - множество отверстий, которые перекрывает корпус (bodyKeepout)
    - дополнительные keepout зоны (для удобства пайки)

### 3) Выбор порядка размещения
- Сначала размещать:
  - компоненты с наибольшим footprint (DIP, разъёмы)
  - компоненты с “жёсткими” ограничениями ориентации/полярности
  - компоненты с высокой связностью по нетлисту (много нетов/пинов)

### 4) Поиск решения (эвристика)
Рекомендуемая стратегия: жадная укладка + локальная оптимизация.

1) Начальное решение:
- ставить каждый компонент в позицию/ориентацию, минимизирующую инкрементальный cost:
  - увеличение bboxArea
  - прирост длины соединений к уже поставленным пинам в тех же net
  - пересечения (штраф)

2) Улучшение:
- локальные “мутации”:
  - сдвиг компонента на ±1..±k по X/Y
  - поворот/смена формы (если допустимо)
  - swap двух компонентов одинакового типа (если помогает)
- принимать улучшения по cost; при застревании допускать редкие ухудшения (simulated annealing-подобно), но без нарушения валидности

### 5) Трассировка (низ платы) и проверка пересечений
- Для каждого net:
  - выбрать “центральный” узел (Steiner-подобно: медиана координат пинов)
  - проложить соединения по Manhattan-маршрутам по сетке
  - если возникает пересечение/конфликт:
    - попробовать альтернативный порядок соединения внутри net
    - попробовать обход
    - если разрешено: предложить джампер (верх) или изолированный провод
- Отдельно обрабатывать шины GND/VCC:
  - если много узлов: предложить “магистраль” вдоль ряда/колонки и отводы

### 6) Вывод результата в файл
Записывать результат в JSON-файл проекта:
- путь: ./layout/perfoboard-layout.json
- если путь недоступен, использовать ./perfoboard-layout.json в корне проекта

Структура JSON:
```json
{
  "version": "1.0",
  "board": { "width": 20, "height": 40 },
  "bbox": { "x1": 5, "y1": 9, "x2": 12, "y2": 14 },
  "components": [
    {
      "ref": "U1",
      "package": "DIP-8",
      "orientation": 0,
      "pins": [
        { "name": "1", "x": 5, "y": 10 },
        { "name": "2", "x": 5, "y": 11 }
      ],
      "keepout": [
        { "x1": 6, "y1": 10, "x2": 7, "y2": 13 }
      ]
    }
  ],
  "nets": [
    {
      "name": "GND",
      "segments": [
        { "x1": 5, "y1": 13, "x2": 9, "y2": 13 },
        { "x1": 9, "y1": 13, "x2": 9, "y2": 10 }
      ],
      "jumpers": []
    }
  ],
  "options": {
    "allowTopJumpers": false
  },
  "cost": {
    "bboxArea": 35,
    "wireLength": 18,
    "crossings": 0
  },
  "metadata": {
    "createdAt": "ISO-8601"
  }
}
```

Требования:
- Координаты целочисленные, соответствуют отверстиям сетки
- Каждый pin имеет уникальную пару (x,y)
- Сегменты нетов идут по манхэттен-маршрутам

## Правила и неизменяемые ограничения

- Каждый вывод вставляется в уникальное отверстие (нельзя делить отверстие между выводами)
- Соединения выполняются пайкой/проводами на нижней стороне между отверстиями (по умолчанию без пересечений)
- Правила конкретных корпусов берутся из project rules-файлов “package_*” в .trae/rules

## Пример входа

```text
Плата: 20x40

Компоненты:
- U1: DIP-8, pins: [1,2,3,4,5,6,7,8]
- R1: AXIAL-0.25W, pins: [1,2]
- C1: RADIAL-2PIN, pins: [+, -]

Неты:
- GND: U1.4, C1.- 
- VCC: U1.8, C1.+ 
- N1: U1.1, R1.1
- N2: U1.2, R1.2
```
